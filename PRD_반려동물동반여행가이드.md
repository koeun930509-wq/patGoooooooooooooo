## 1. 배경 및 문제 정의

반려가구가 국내 여행·나들이를 갈 때 가장 큰 불편은 "이 장소에 우리 아이를 데려가도 되는가?"에 대한 정보가 **모호하고 파편화**되어 있다는 점이다.

현장에서 실제로 겪는 문제:

- **출입 조건이 애매하게 표기됨** — "반려동물 동반 가능"이라고만 적혀 있으나, 실제로는 "실외 테라스만", "소형견만", "맹견은 입마개 필수" 등 세부 조건이 붙는 경우가 많다.
- **규정 인지 오류로 현장 입장 거부** — 대형견/맹견이라 거부당하거나, 이동장·목줄·입마개 등 필수 준비물이 없어서 되돌아온다.
- **헛걸음 발생** — 왕복 이동 후 입장 불가를 확인 → 시간·비용·감정 손실.

즉, 데이터는 존재하지만(공사 API의 `detailPetTour2` 등) **개별 반려동물 특성에 맞춰 해석·판정해 주는 서비스가 없다.** 이 갭을 메우는 것이 본 서비스의 목적이다.

## 2. 목표 (Goals) 및 성공 지표

### 2.1 제품 목표
1. 반려동물 프로필과 목적지 규정을 대조해 **"동반 가능 / 조건부 가능 / 불가"를 명확히 판정**한다.
2. 판정과 함께 **개별 반려동물에 맞춘 필수 준비물 체크리스트**를 제공해 현장 거부·헛걸음을 예방한다.
3. 모호한 원문 규정을 **사람이 바로 이해할 수 있는 문장**으로 재해석해 준다.

### 2.2 성공 지표 (KPI 예시)
| 지표 | 목표 |
|---|---|
| 판정 정확도(원문 대비 오판정률) | ≤ 5% |
| "헛걸음 방지" 체감 만족도(설문) | ≥ 4.2 / 5.0 |
| 목적지 조회 → 판정 확인 전환율 | ≥ 60% |
| 준비물 체크리스트 사용률 | ≥ 40% |

## 3. 타깃 사용자 & 페르소나

**타깃**: 반려동물과 함께 국내 여행·나들이를 안전하게 즐기려는 반려가구 여행객.

| 페르소나 | 상황 | 니즈 |
|---|---|---|
| 주말 나들이형 (소형견 보호자) | 근교 카페·공원을 즉흥적으로 찾음 | 지금 위치 주변에 갈 수 있는 곳 + 준비물 |
| 여행 계획형 (중·대형견 보호자) | 1박 이상 여행을 사전 계획 | 목적지별 동반 가능 여부를 미리 확정하고 싶음 |
| 맹견/특수견 보호자 | 입장 거부 경험이 많음 | "우리 아이가 정말 들어갈 수 있는지"의 확실한 답 |

---

## 4. 핵심 가치 제안 & 차별점

- **"조건 해석" 엔진**: 단순 목록 노출이 아니라, 내 반려동물 특성(견종·무게·맹견 여부 등)과 장소 규정을 대조해 **판정**해 준다.
- **헛걸음 방지 중심 UX**: 결과 화면의 최상단이 "데려갈 수 있음/없음 + 이유"다.
- **맞춤 준비물 안내**: 장소 규정에서 요구하는 준비물(목줄 길이, 입마개, 배변봉투, 이동장 등)을 반려동물별로 자동 산출.

## 5. 서비스 범위 (MVP Scope)

### In-scope (MVP)
- 반려동물 프로필 등록/설정
- 목적지 탐색(지역 기반 / 내 주변 / 키워드)
- 장소별 동반 조건 조회 및 **AI 판정 + 준비물 체크리스트**
- 장소 상세(주소·운영시간·주차·전화 등)

### Out-of-scope (차기)
- 예약/결제, 실시간 혼잡도, 커뮤니티/후기, 지도 경로 최적화, 다국어

## 6. 기능 요구사항 (Functional Requirements)

과제에 명시된 두 기능을 세분화한다.

### FR-1. 반려동물 세부 특성 및 목적지 설정
| ID | 요구사항 |
|---|---|
| FR-1.1 | 반려동물 프로필 입력: 이름, 종(개/고양이/기타), 견종, **무게(kg)**, 크기(소/중/대형), 맹견 여부, 입마개 착용 가능 여부, 예방접종/중성화 여부(선택) |
| FR-1.2 | 프로필 다중 등록(여러 마리) 및 이번 여행 동반 대상 선택 |
| FR-1.3 | 목적지 설정 3방식: (a) 지역 선택 `areaBasedList2`(법정동 코드 `ldongCode2` 활용), (b) 내 주변 `locationBasedList2`(GPS 반경), (c) 키워드 검색 `searchKeyword2` |
| FR-1.4 | 관광타입 필터(`contentTypeId`: 12 관광지·14 문화시설·15 행사·28 레포츠·32 숙박·38 쇼핑·39 음식점) |

### FR-2. 출입 규정 분석 기반 동반 가능 여부 및 필수 준비물 안내
| ID | 요구사항 |
|---|---|
| FR-2.1 | 장소 선택 시 `detailPetTour2` 호출로 동반 조건 원문 취득 |
| FR-2.2 | 반려동물 프로필 ↔ 장소 규정 대조 후 **판정**: `가능` / `조건부 가능` / `불가` |
| FR-2.3 | 판정 근거를 자연어로 제시(예: "맹견이므로 입마개 착용 시 동반 가능") |
| FR-2.4 | 반려동물 맞춤 **필수 준비물 체크리스트** 산출(목줄·입마개·배변봉투·이동장 등) |
| FR-2.5 | 사고 대비/주의사항(`relaAcdntRiskMtr`, `etcAcmpyInfo`) 요약 표시 |
| FR-2.6 | 장소 상세(주소·운영시간·주차·전화)는 `detailCommon2` / `detailIntro2` 병합 표시 |
| FR-2.7 | 원문 규정 전문(펼쳐보기)을 제공해 신뢰성 확보 |

## 8. 조건 해석 로직 (판정 규칙)

원문 필드는 자유 텍스트라 규칙 기반만으로는 한계가 있으므로 **규칙 + LLM 해석** 하이브리드로 설계한다.

### 8.1 1차 규칙 필터 (결정적)
- `acmpyPsblCpam`에 "소형견"만 명시 & 반려동물 무게 > 10kg → **불가/조건부**
- "맹견" 관련 조건 존재 & 맹견 여부 = true → 입마개 필수 조건 부여
- `acmpyTypeCd`가 "일부구역/실외만" → **조건부 가능**(구역 제한 명시)
- 필드가 비어 있음("") → "정보 없음"으로 표기하고 임의 단정 금지

### 8.2 2차 LLM 해석 (모호성 처리)
규칙으로 확정되지 않는 자유 텍스트를 LLM이 해석해 다음을 산출한다.
1. 판정: `가능` / `조건부 가능` / `불가` / `정보 부족`
2. 판정 근거 1~2문장
3. 필수 준비물 체크리스트(반려동물 특성 반영)
4. 주의사항 요약

> 상세 프롬프트는 별도 파일 `개발프롬프트.md`의 "조건 해석 엔진" 섹션 참조.

### 8.3 신뢰성 원칙
- **추측 금지**: 원문에 없는 조건을 만들어내지 않는다. 정보가 없으면 "현장 확인 권장"으로 안내.
- **원문 병기**: 모든 판정 옆에 원문 규정 펼쳐보기를 제공한다.

[프로필 등록] → [목적지 설정: 지역/내주변/키워드] → [장소 목록]

   → (선택) 원문 규정 펼쳐보기 / 장소 상세(주소·운영시간·주차)

## 10. 화면 구성 (IA)

1. **온보딩/프로필**: 반려동물 특성 입력(무게 슬라이더, 맹견 토글 등)
2. **탐색**: 탭(지역 / 내 주변 / 검색) + 타입 필터, 리스트/카드
3. **판정 상세(핵심 화면)**:
   - 상단 배지: 🟢 가능 / 🟡 조건부 / 🔴 불가 / ⚪ 정보 부족
   - 판정 근거 문장
   - 준비물 체크리스트(체크 가능)
   - 주의사항 카드
   - 원문 규정(접힘) + 장소 기본정보

## 12. 기술 스택 제안 (MVP)

- **프론트엔드**: React + TypeScript (모바일 웹 우선, PWA 선택)
- **백엔드(BFF)**: Node.js/Express 또는 Next.js API Route — TourAPI 프록시·캐싱·키 보호
- **해석 엔진**: LLM API(예: Claude) — 규칙 필터 후 모호 케이스만 호출해 비용·쿼터 절감
- **캐시/DB**: Redis 또는 파일/메모리 캐시(MVP), 콘텐츠 메타 SQLite/Postgres
- **지도**(선택): 좌표(`mapx`,`mapy`) 기반 지도 표출
